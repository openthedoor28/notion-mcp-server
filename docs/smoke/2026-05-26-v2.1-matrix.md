## Findings (no fix yet)

### Atomic + concurrency > 1 is racy
`{ atomic: true, concurrency: 3, items: [ok, fail, ok] }` returned `succeeded: 2,
failed: 1` instead of aborting before the third item. Root cause: the `aborted`
flag in `runBatch` only guards future dispatches, but with `concurrency: 3` all
three workers start immediately. Documented contract (`BATCH_ENVELOPE_HELP`) says
"best-effort rollback of created entities on first failure" ‚Äî concurrency
behavior isn't pinned in docs.

- `{atomic: true, concurrency: 1}` with `set_page_title` correctly halts at item 1
  and skips item 2 with `code: "aborted"`.
- `{atomic: true, concurrency: 1}` with `create_page` correctly rolls back the
  successful index 0 (verified `in_trash: true`) and skips index 2. Report
  includes `rolled_back: 1`.

Suggested follow-up: either force `concurrency = 1` when `atomic: true`, or
document that atomic mode only abort-blocks queued items.

### `get_comment` slim shape inconsistent
`add_page_comment` / `list_comments` slim view includes `text`, `parent`,
`discussion_id`, `created_by`, `created_time`. `get_comment` slim drops everything
except `id` and `created_time`. `verbose: true` confirms the raw API response is
complete. Slim handler should match the others.

### `position: "start"` on `append_blocks` returns the full child list
With 11 children, prepending 1 block returned `appended: 11, results: [<all 11>]`.
The Notion API behavior is to return the updated child set, not the newly added
ones. We report `response.results.length` verbatim, which is misleading.

Suggested fix: trust `children.length` (or `parseMarkdownToBlocks(markdown).length`)
for the `appended` count instead of `response.results.length`.
